# Production readiness checklist

Use this checklist for a native Klein streaming job before it receives
production traffic. A bounded job lowered to Ray Data follows Ray Data's
execution, retry, and writer contracts instead; first confirm the selected mode
after registering all terminals with `ray.klein.explain("<job-name>")` and the
[operator compatibility matrix](operator-compatibility.md).

:::{warning}
Klein is alpha software. Passing this checklist does not turn an unsupported
version, connector, topology change, or multi-tenant deployment into a
supported one. The current compatibility target is Python 3.10--3.12 with
`ray>=2.56.1,<2.57`; see [Compatibility](compatibility.md).
:::

Before reviewing individual controls, write down the job's recovery point
objective (RPO), recovery time objective (RTO), maximum acceptable duplicate
window, output-visibility latency, and peak-load target. A configuration is
ready only when those targets have been tested with the real source, state
shape, and sink behavior.

## Review the job

In the table, *Expected* is the release gate. Record concrete evidence rather
than checking a box from configuration alone.

| Area | Verification | Expected | Risk | Action |
| --- | --- | --- | --- | --- |
| Supported runtime | Print `sys.version`, `ray.__version__`, and `ray.klein.__version__` in the submission process and representative worker pools. Compare the installed extras and native libraries with the immutable image or lock file. | Python is 3.10--3.12, Ray is at least 2.56.1 and below 2.57, and every eligible worker has the same Klein artifact and connector extras. | A worker can fail only when a task or connector is first materialized; a different Ray minor may change DeveloperAPI behavior. | Pin the complete environment and image digest. Rebuild it and run integration tests; do not widen the Ray range only in deployment metadata. |
| Execution mode and graph | Set `execution.runtime.mode` deliberately for a production job. After registering all terminals, save the output of `ray.klein.explain("<job-name>")` with the release artifact and review the operator IDs, concurrency, resources, partition edges, sources, and sinks. | The job selects the intended runtime, and the reviewed plan matches the submitted artifact. | `auto` can select batch after a graph edit; construction-order changes can alter the numeric identities used by checkpointed source and operator state. | Correct the graph or select `streaming`. Treat the saved plan as upgrade evidence; follow [Upgrade Klein jobs](upgrading.md) before changing it. |
| Remote cluster and packaging | Confirm the entry point calls `ray.init(address="auto")` or an approved remote address before execution. Submit a smoke task to each worker pool and import every optional connector it may run. | No production job silently creates an embedded local Ray cluster, and imports succeed on every eligible node. | Work can run on the wrong cluster or fail during recovery when Ray schedules a replacement on a differently built node. | Use an immutable Ray runtime environment or identical head/worker images. Follow [Deployment](deployment.md). |
| Capacity and placement | Compare the saved plan with Ray cluster capacity. During a peak-load test, watch input-buffer rows and bytes, replay-buffer bytes, Object Store pressure, backpressure duration, CPU/GPU use, and placement fallback warnings. | All live operators, the JobManager, checkpoint coordinator, and optional Serve actors fit with failure headroom; buffers remain bounded and checkpoints still complete at peak load. | Gang scheduling can time out, fallback placement can change locality, and replay or Object Store pressure can fail tasks or amplify recovery time. | Add resources or reduce concurrency/buffer limits. Tune from measured symptoms using [Performance tuning](performance-tuning.md), not by removing bounds. |
| Job identity | Record the Ray Jobs submission ID, Klein job name, returned `handle.namespace`, source consumer identity, artifact digest, and checkpoint root. If `job.namespace` is explicit, check that no old actors remain before reusing it. | Operations can unambiguously address one job and can reproduce the submitted definition. | Reusing a live namespace can attach to or control the wrong detached actors; an autogenerated namespace can be lost after the submitting process exits. | Persist deployment metadata outside the Ray cluster. Use a stable namespace only with a controlled stop/redeploy procedure. A Ray namespace is not a security boundary. |
| Durable checkpoint location | Inspect `execution.checkpointing.dir`. From every worker security domain, test read/write access using the same workload identity. Verify that a completed `chk-N` contains `_metadata`. | The URI is shared durable storage (`s3://`, `gs://`, or an equivalently durable supported filesystem), not a node-local temporary path. `_metadata` is readable after completion. | Local disk and the Ray Object Store cannot recover a lost node or cluster. Partial checkpoint directories without `_metadata` are not restorable. | Configure a least-privilege durable prefix and validate it with [Checkpoint storage](checkpoint-storage.md). |
| Checkpoint cadence and retention | Review `execution.checkpointing.trigger.interval-duration`, `execution.checkpointing.trigger.interval-records`, timeout, concurrency, persistence interval, and retained count. Observe multiple completions under normal and peak load. | At least one trigger is nonzero; the measured interval meets the RPO and sink-visibility target; timeout exceeds observed worst-case duration; policy normally retains at least two usable revisions. | Setting both triggers to `0` prevents periodic recovery points and delays transactional sink visibility. A single bad retained checkpoint can remove the rollback path. | Choose thresholds from the SLA, increase retention, and alert on age, duration, and consecutive failures. Keep storage lifecycle rules longer than the required rollback window. |
| State backend and key groups | Record `state.backend.type`, `state.backend.local-dir`, `state.keyed.max-parallelism`, state descriptor names, serializers, and the largest keyed-operator concurrency. Exercise a restore with production-shaped state. | The backend dependency is installed; RocksDB has adequate node-local disk; max parallelism is at least current concurrency and remains identical across restore; descriptor and serialized value shapes are stable. | Changing max parallelism prevents keyed-state restore. Changing backend or serialized state has no documented automatic migration path. Node-local RocksDB alone is not durable. | Freeze these values with the release. Use completed durable checkpoints for recovery and test any state change as an upgrade. See [Managed state](ray-native-state.md). |
| Source recovery | For every source, identify its authoritative recovery position and run a worker-kill and whole-job restore test. Compare the position before failure with the resumed position. | The observed replay/loss envelope matches the source row in [Delivery guarantees](delivery-semantics.md). Kafka resumes from checkpointed offsets; custom source state represents the next read position. | RocketMQ progress is broker-managed and can move ahead of a Klein checkpoint. A custom source can silently restart from the wrong position. | Select a source compatible with the SLA, implement idempotent checkpoint completion, or add upstream replay/archival. Do not infer exactly-once from managed state alone. |
| Sink effects | Kill a sink after its external write and before checkpoint completion, then recover. Inspect duplicates, staged objects, transactions, and visibility time. | The tested effect matches the documented sink guarantee. Filesystem/Iceberg/two-phase sinks publish only after checkpoint durability; other built-in streaming sinks are generally at-least-once. | A nontransactional sink can expose a write that is replayed. A malformed two-phase implementation can double-commit or leak prepared work. | Use a stable idempotency key, database upsert, downstream deduplication, or a correctly implemented `TwoPhaseCommitSinkFunction`. Document reconciliation. |
| Restart policy and recovery | Review fixed-delay restart attempts, delay, count window, health-check interval, and deployment/RPC timeouts. Kill a transform worker, a stateful worker, a sink worker, and the checkpoint coordinator while data is flowing. | Each injected failure either recovers within the RTO from replay/checkpoint state or reaches a visible terminal failure after the configured retry budget. A fresh checkpoint completes after recovery. | Excessive retries can hide a permanent defect; a retry budget that is too small can fail on transient infrastructure loss. | Set a bounded policy, alert on restart rate and terminal status, and retain the first exception and checkpoint history for diagnosis. |
| Driver, JobManager, and cluster loss | Exit the submitting driver without cancellation, then query the job from another process. Separately rehearse loss of the JobManager process and of the Ray cluster. | Driver exit leaves the detached job running. Loss of the JobManager process or whole cluster is recovered by a controlled resubmission from an explicit completed checkpoint within the RTO. | Detached actors are not cluster durability. Current actor constructor state cannot reconstruct a submitted graph after JobManager process loss. | Store the entry point, configuration, namespace, and checkpoint URI outside the cluster. Rehearse [checkpoint recovery](checkpoint-recovery.md) and [driver failure](driver-fault-tolerance.md). |
| Event time and idle inputs | Feed delayed records and leave one physical input partition idle. Observe input/output watermark, watermark lag, idle inputs, late records, and window output. | Watermarks advance according to the application policy, idle inputs do not stall progress indefinitely, and the accepted late-data policy is measured and documented. | A timestamp selector without watermarks never closes event-time windows; an incorrect idle timeout can mark a slow partition idle and classify later records as late. | Configure and test a `WatermarkStrategy`, idleness, allowed lateness, and TTL using [Event time](event-time.md). |
| Observability and alerting | From a second process, run `ray-klein list`, `ray-klein status <namespace> --json`, and `ray-klein attach <namespace>`. Confirm Ray log retention and scrape the documented job/task/operator metrics. Force one representative alert condition. | Operators can distinguish running, recovering, stale, failed, and cancelled states; alerts cover checkpoint age/failures, restart rate, source lag, watermark lag, backpressure, state/replay growth, sink failures, and lost scrape targets. | The Dashboard state actor keeps only in-memory history; a green process metric does not prove data is progressing; `dashboard_stale` can represent cached state. | Export Ray metrics and logs to durable systems. Define runbooks using [Observability](observability.md) and [Troubleshooting](troubleshooting.md). |
| Security boundary | Review the deployment against [Security](security.md). Confirm Ray authentication/network controls, least-privilege checkpoint and connector identities, secret delivery, and the Dashboard listener. | Only trusted code can submit to the Ray cluster; mutually untrusted workloads use separate clusters; the Klein Dashboard remains loopback-only or sits behind TLS and authentication. | Klein executes UDFs and restores pickle payloads. The Dashboard control endpoint has no built-in authentication. Namespaces do not isolate hostile tenants. | Fix the platform boundary before launch. Never use `--allow-unauthenticated` as the only protection for a remote listener. |
| Upgrade and rollback | Rehearse the exact candidate and rollback artifacts with production-shaped state in an isolated environment. Record an old-version completed checkpoint before the change and prove that the old artifact can restore it. | The candidate restores, processes representative traffic, produces the expected effects, and completes a new checkpoint; rollback remains possible from the recorded pre-upgrade checkpoint. | Alpha checkpoints have no cross-release migration promise, and a checkpoint written by the candidate may not be readable by the old version. Restoring an older point may replay external effects. | Follow [Upgrade Klein jobs](upgrading.md); retain the old artifact, dependency lock, configuration, checkpoint prefix, and deduplication/reconciliation procedure. |

## Run a failure rehearsal

Run the following sequence in an environment with production-equivalent Ray
settings and isolated external sinks:

1. Start at normal load and wait for at least two completed checkpoints.
2. Kill one transform worker while its output and replay buffers are nonempty.
3. Kill a stateful worker after it has accumulated representative state.
4. Interrupt a sink between its external write and checkpoint completion.
5. Exit the submitting driver and operate the detached job from another client.
6. Stop the cluster, recreate it, and resubmit the same graph with
   `execution.savepoint.path` set to the recorded completed `chk-N` URI.
7. Verify source position, state contents, watermarks, output duplicates or
   transactions, and completion of the first post-restore checkpoint.

Do not use `ray-klein stop` as a checkpoint operation. It requests
cancellation and does not create a named savepoint. Wait for and record an
already completed checkpoint before the planned stop.

## Record the release decision

Keep this evidence with each production release:

- artifact and worker-image digests, dependency lock, Python/Ray/Klein versions;
- saved logical/resource plan and canonical configuration with secrets removed;
- job name, namespace, source identities, sink destinations, and checkpoint root;
- SLA targets and peak-load measurements;
- failure-injection results and accepted duplicate/loss envelope;
- latest tested restore checkpoint, restore duration, and rollback checkpoint;
- owners for application, Ray cluster, storage, connectors, and incident response.

A deployment is not ready when a required row is merely “accepted risk” with
no owner, expiry, detection signal, and recovery action.
